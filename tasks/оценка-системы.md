# Оценка системы AI Corp и пробелы для полной автоматизации

## Текущий уровень: продвинутый прототип (MVP+)

Система уже делает впечатляющий объём работы: 50 инструментов для агентов, 4 способа триггера (Telegram, cron, автопилот задач, меж-агентские сообщения), трехуровневая система личности (SOUL.md, IDENTITY.md, ROLE.md), Docker-песочница для команд, событийная система, темплейты компаний. Это сильный фундамент.

**Сильные стороны:**
- Богатый набор инструментов (50) — агенты могут управлять задачами, файлами, ролями, кронами, сообщениями, веб-поиском
- 4 канала запуска: Telegram-бот, cron-задачи, автопилот In Progress, меж-агентские сообщения
- Ролевая система с glob-правами на файлы — гибкий контроль доступа
- Иерархическая (parentId) и peer-to-peer (collaborators) модель связей агентов
- Персистентная память с LLM-суммаризацией и retrieval (keyword-based)
- Docker-песочница для выполнения команд с policy engine (allow/deny/approve)
- Трёхуровневая система личности: ROLE, IDENTITY, SOUL — автогенерация по роли + кастомизация
- Kanban-доска, полностью интегрированная с автономией агентов
- Telegram-боты per-agent (каждый агент может иметь своего бота)
- Учёт стоимости LLM-вызовов (токены, модель, цена)
- Веб-поиск (Brave API или self-hosted SearXNG)

## Что не хватает для полной автоматизации — по слоям

### 1. Планирование и декомпозиция целей

- **Нет декомпозиции высокоуровневых целей.** Агент получает задачу, но не умеет сам разбить большую цель на дерево подзадач, оценить их, расставить приоритеты и зависимости.
- **Нет зависимостей между задачами (`dependsOn`).** Агент не знает, что "задача Б заблокирована, пока не готова задача А".
- **Нет автоматического планирования спринта.** Никто не берёт задачи из Backlog в работу по приоритету — только ручное перетаскивание.
- **Нет due dates / дедлайнов.** Задачи не имеют временных ограничений, агенты не могут приоритизировать по срочности.

### 2. Оркестрация командной работы

- **Нет коллективных обсуждений.** `send_message` и `ask_agent` работают только 1:1. Нет групповых чатов агентов для совместного принятия решений.
- **Нет workflow/pipeline между агентами.** Например: Dev сделал → Reviewer проверил → DevOps задеплоил. Сейчас это надо настраивать вручную через crons и события.
- **Нет механизма консенсуса/голосования** — когда несколько агентов должны принять совместное решение.
- **Нет делегирования с контролем.** Manager не может поручить подзадачу и получить уведомление о завершении с автоматической проверкой качества.

### 3. Событийная система (всего 4 типа событий)

- **Все события только про задачи:** `task.status.changed`, `task.completed`, `task.comment.added`, `task.assignee.changed`.
- **Не хватает событий:**
  - `agent.status.changed` — агент перешёл в Idle / Working / Error
  - `file.changed` — файловый вотчер в workspace
  - `webhook.received` — внешний триггер
  - `approval.resolved` — одобрение получено или отклонено
  - `message.received` — агент получил сообщение
  - `cron.completed` / `cron.failed` — результат cron-задачи
  - `workspace.settings.updated` — изменения в настройках
- **Нет внешних триггеров (webhooks).** Нельзя запустить агента по GitHub webhook, CI-событию, HTTP-запросу извне.
- **Нет файлового вотчера.** Агент не узнает, что кто-то изменил файл в workspace без постоянного опроса.

### 4. Интеграции с внешним миром

- **Нет generic HTTP-клиента.** Агент не может вызвать произвольное API (Stripe, GitHub, Slack, Jira). Есть только `fetch_url` — read-only HTML.
- **Нет email (SMTP).** Агент не может отправить отчёт по почте.
- **Нет работы с БД.** Агент не может выполнить SQL-запрос к внешней базе.
- **Нет Git-native инструментов.** Нет `git status`, `git diff`, `git log`, `create PR` как инструментов первого класса. Всё через сырой `run_command` в песочнице.
- **Нет календаря.** Нет due dates у задач, нет планирования встреч.

### 5. Обучение и самоулучшение агентов

- **Нет цикла обратной связи.** Результаты работы агента не оцениваются — нет метрик успеха/провала задачи, на основе которых агент мог бы корректировать поведение в будущем.
- **Память не семантическая.** Retrieval чисто keyword-based (частотный индекс), а не векторный/эмбеддинговый поиск. Агент не найдёт релевантный прошлый опыт по смысловому сходству.
- **Нет cross-agent knowledge sharing.** Каждый агент полностью изолирован в своей памяти, не может поучиться у коллеги или использовать чужой опыт.
- **Нет A/B тестирования промптов.** Нельзя сравнить две версии SOUL.md по объективным метрикам производительности.

### 6. Безопасность и контроль при автоматизации

- **API без аутентификации.** Любой с доступом к порту 4000 может управлять агентами. Критично для production.
- **Нет rate-limiting на LLM-вызовы.** Агент в бесконечном цикле tool calling может сжечь бюджет за минуты.
- **Нет cost budgets.** Нет квот на workspace или агента ("не трать больше $5 в день", "не превышай 100K токенов").
- **Approve-ы не эскалируются.** Если запрос на одобрение висит 3 дня — никто не узнает, задача заморожена навсегда.
- **Нет audit trail.** Кто, что и когда менял — только capped 100 записей лога, нет неизменяемого журнала.

### 7. Эксплуатационные (production-ready) пробелы

- **JSON-хранилище без конкурентного доступа.** Race condition при одновременной записи из cron + telegram + autopilot. Данные могут быть потеряны.
- **Нет мониторинга здоровья агентов.** Если агент в статусе Error — нет алерта, никто не узнает.
- **Нет бэкапа/восстановления** состояния агентов и workspace.
- **Нет миграций схемы данных.** При изменении структуры типов старые JSON-файлы могут стать несовместимыми.
- **Telegram — polling, не webhooks.** Не масштабируется на большое количество ботов. Нет rich media (только текст, без изображений и файлов).
- **Нет graceful shutdown.** При перезапуске сервера все выполняющиеся сессии агентов теряются.

### 8. Автопилот задач — ограничения

- **Только для `In Progress`.** Агент не умеет сам брать задачи из Backlog по приоритету и двигать в работу.
- **Один агент — одна задача.** Не может работать над несколькими задачами параллельно.
- **Hard-cap 40 итераций** — сложная задача (например, полный рефакторинг модуля) может не уложиться в лимит.
- **Нет стратегии повторных попыток при ошибках (retry).** Ошибка в одном tool call — автопилот падает полностью вместо повтора с другим подходом.
- **Нет Pause/Resume для автопилота.** Только полная остановка при `request_approval`.

### 9. Отсутствующие инструменты агента

- **Git-native:** `create_pr`, `git_diff`, `git_log`, `git_blame` — вместо сырого `run_command("git ...")`
- **HTTP:** `http_request(method, url, headers, body)` — generic HTTP-клиент
- **Email:** `send_email(to, subject, body)` — SMTP-отправка
- **Базы данных:** `run_sql(query, params)` — выполнение SQL
- **Календарь:** `schedule_meeting`, `check_calendar` — работа с календарём
- **Webhooks:** `invoke_webhook(url, payload)` — вызов внешнего webhook
- **Тестирование:** `run_tests`, `evaluate_code` — запуск тестов и анализ результатов (сейчас только `run_command`)

### 10. Тестирование и симуляция

- **Нет sandbox-режима.** Нельзя проверить, как агент поведёт себя с новым SOUL.md или промптом, не рискуя реальными данными.
- **Нет regression-тестов для поведения агентов.** Изменение промпта может сломать цепочку reasoning, и это не будет обнаружено.
- **Нет evaluation framework.** Нельзя измерить качество работы агента на наборе тестовых сценариев.
- **Минимальное покрытие тестами.** Только 2 файла: `glob.test.ts` и `telegram-formatter.test.ts`.

## Приоритизированный roadmap

| Приоритет | Что | Почему |
|-----------|-----|--------|
| **P0** | API authentication / авторизация | Безопасность — нельзя в production без защиты API |
| **P0** | Атомарное хранилище (SQLite вместо JSON) | Race conditions в JSON-файлах ломают данные при concurrent-доступе |
| **P1** | Зависимости задач (`dependsOn`, `blocks`) | Базовая orchestration невозможна без понимания зависимостей |
| **P1** | Декомпозиция целей агентом (goal → task tree) | Ключевой шаг к автономному планированию |
| **P1** | Webhook-триггеры и внешние события | Интеграция с реальным миром (GitHub, CI/CD, мониторинг) |
| **P1** | Cost budgets + rate limiting для LLM | Защита от runaway-агентов, контроль расходов |
| **P2** | Multi-agent workflows (Dev → Review → Deploy pipeline) | Сквозная автоматизация типовых процессов разработки |
| **P2** | Семантическая память (embeddings) | Качественный retrieval опыта по смыслу, а не по ключевым словам |
| **P2** | Generic HTTP tool (`http_request`) | Интеграция с любым внешним API |
| **P2** | Git-native инструменты | Стандарт для dev-агентов, безопаснее чем сырой `run_command` |
| **P3** | Групповые обсуждения агентов | Коллективное принятие решений |
| **P3** | Эскалация approve-ов (timeout → alert) | Чтобы одобрения не зависали навсегда |
| **P3** | Мониторинг здоровья агентов / алерты | Production readiness |
| **P3** | A/B тестирование промптов | Итеративное улучшение поведения агентов |
| **P3** | Evaluation framework для агентов | Измеряемое качество, защита от регрессий |

## Итоговая оценка

**Система на ~60-65% пути к полной автономии агентов.**

Сильные стороны: богатый набор инструментов, гибкая система ролей и прав, несколько каналов запуска агентов, контекстная память с суммаризацией.

Ключевые блокеры для production: JSON-хранилище с race condition, отсутствие аутентификации API, отсутствие планирования/декомпозиции целей агентами, нет workflow-оркестрации между агентами, всего 4 типа событий без внешних триггеров, нет механизма самообучения агентов.
